Birthday Paradox Simulator

The "birthday paradox" is a probability problem that says in a group of just 23 people, there is a greater than 50% chance that at least two people share the same birthday. The paradox arises because people tend to underestimate how quickly the probability of having a shared birthday increases as the size of the group grows.

The goal of this project was to write code to test the birthday paradox through a series of experiments. The application runs experiments with different group sizes to demonstrate how rapidly the likelihood of shared birthdays increase with group size. This project is made up of several parts. Simulation.java runs the simulation by creating Experiment objects of different group sizes (5-100 people) and displays the percentage of trials with shared birthdays. Experiment.java represents a single experiment and calculates the percentage of trials (20 per group size) that resulted in at least one shared birthday. Trial.java represents a single trial within an experiment and creates a certain number of people (with random birthdays) and determines whether any two people in the group share a birthday. Birthday.java represents one person's birthday (month and day) and accounts for different month lengths. The Birthday class generates random, valid birthdays to determine if two are the same. These classes work together in a hierarchy to bring this probability theory to life. Simulation creates multiple Experiments, each Experiment runs multiple Trials, and each Trial creates multiple Birthday objects to simulate groups of different sizes. This simulator provides valuable insights into this famously counterintuitive probability theory. This simulator provides valuable insights into this famously counterintuitive probability theory.

In implementing the Birthday Paradox simulator, I focused on developing the functionality within the class structure that my partner had worked to establish. My main goal was working to help implement the logic that made each class behave together properly and function successfully. When it came to developing tests for these classes, I worked mainly on testing the Birthday class and the Experiment class. To do this, I implemented Mockito to create mock objects that simulated a variety of different scenarios without relying on randomness, which helped me achieve consistent and reliable test results. Throughout the project, my partner and I utilized pair programming which helped improve the quality of our code and our understanding of the application significantly.

Throughout this project, I significantly strengthened my Java testing skills, specifically in using mocking to create reliable test environments for processes that are normally random. Elevator Simulator

The Elevator Simulator project involved writing code to develop ad elevator simulator that models how different types of elevators operate in a building. Each elevator has a unique set of rules including what floors they service and how many passengers can be on an elevator at a time. A scheduler assigns the calls to the best elevator available. The simulator moves the elevators on each round and shows what elevators are moving, stopping, or waiting and how many people are on each. This project helps to understand how an elevator system functions in real life.

The goal of this project was to write code to simulate how different types of elevators function on buildings with multiple floors using Test Driven Development, maintaining best object-oriented practices, SOLID principles, and clean code in mind. The project included 4 different types of elevators: Standard Elevators, Freight Elevators, Pent House Elevator, and Express elevator. Each elevator has specific policies that must be kept including how many passengers it can have, when it can accept calls, what floors it can travel to. For example the Penthouse elevator only travels between the first floor and the top floor. The express elevator skips the middle floors. The scheduler assigns new calls to the appropriate elevator based on their current state which includes their position, direction, and occupancy. The design of the code follows OOP principles, with a base elevator class that defines the common behavior of all elevators. The ElevatorFactory creates different elevator types using the factory pattern with specialized subclasses for each specific elevator type and their unique behaviors. The Scheduler uses a Strategy design pattern with SimpleScheduler and ComplexScheduler implementations with different algorithm to assign calls to elevators. The Simulator controls the program, it initializes the program with 7 command-line arguments. It creates elevators, and runs the simulation in rounds. Each round involves assigning calls to the elevators and moving the elevators and continues until all the calls are handled and each of the elevators have returned to the first floor.

In implementing the Elevator simulator, I focused on developing the Elevator abstract class and the specific subclasses. I used Test Driven development to first work on the implementing the abstract Elevator. After I finished with the base Elevator I was able to create the specific subclasses and used override for how the different elevators were able to choose when the accepted calls on not based on their specific qualities. This structure made it easier to expand and test each of the different elevator types individually.

Throughout this project, I strengthened my Test Driven Development Skills by writing tests before implementing the code. This strategy helped me to debug my code as I was working on it and develop each part of my program separately. My partner and I utilized pair programming which helped both of us give good feedback and constructive comments on each other code and understand the parts of the code each other had written.

Pizza Ordering System

For our final project for COMP 305 my group wanted to choose a project that would challenge our abilities to use and implement different design patterns we learned about in class. We wanted to create a project that allows users to create standard pizzas or make custom pizzas with a variety of toppings while maintaining a clean, and organized structure. This project gave us the opportunity to challenge ourselves to implement the Decorator design pattern in Java, which we had not learned how to do in class.

The Pizza System allows users to choose to either order a Standard Pizza with designated cheese, sauce, and crust's or create a Custom Pizza. If a user selects to create a custom pizza the user is able to customize the pizza by selecting from various types of cheeses, sauces, crusts, and toppings. We wanted this ordering system to be flexible and customizable: - Start by deciding what type of Pizza you want: (Neapolitan, New York, Chicago, or Custom) - If user selects custom: they get to customize their pizza by choosing from the different types of Sauce, Cheese, and Crust - Add multiple toppings which update cost calculation - Get a detailed description of your pizza - Choose a preferred payment method (cash or card) Our Pizza project calculates the costs of your selected pizza's ingredients and generates a description of your delicious pizza including its name, ingredients and any added toppings, and then the user checkouts with their preferred payment method(cash or card).

Our project implements multiple design patterns so that our project architecture is flexible and maintainable.

Decorator Pattern

Our pizza program uses the Decorator pattern for adding toppings, which allows for: Flexibility: Can handle unlimited topping combinations Maintainable: Separates the core pizza functionality from the optional added toppings Extension: We can easily add new toppings in the future without modifying existing code

Simple Factory Pattern

Pizza Factory implements the Simple Factory pattern to create standard pizza styles Encapsulation: Hides the complexity of creating different styles of pizza Single Responsibility Principle: Keeps pizza creation logic in one place Consistency: Ensures pizzas are created with the correct default ingredients

Strategy Pattern

To deal with payment processing, we implemented the Strategy pattern: Flexibility: Allows users to choose between payment methods (cash or credit) Extension: Easy to add new payments in the future Testability: Able to test payment strategies individually

We chose Java for implementation because it is effective for using object-oriented programming to implement design patterns. Java uses interfaces and inheritance to emphasize clear relationships between components, allowing us to make our code maintainable and readable.

The overall design of our project includes: Pizza interface: Defines core functionality. SimplePizza: Concrete implementation of Pizza interface. PizzaFactory: Creates standard pizza styles (New York, Chicago, Neapolitan, and Custom) Toppings: Decorator hierarchy that wraps pizzas with additional ingredients Payment Strategies: implement common interface for different payment methods (cash, card)

I worked on implementing the core architecture of our project. I focused on: The Pizza interface and the SimplePizza concrete class. The Toppings decorator base class and the different concrete toppings. The Pizza Factory for creating standard styles of pizza.

I implemented these features following test-driven development, by writing tests before implementing each feature to ensure they were correct before moving on. I have never implemented the Decorator Pattern officially before so this project was a great learning experience for me where I got to challenge my abilities to implement different design patterns together.
